### Critical Areas Requiring Deep Analysis
1. **Transaction Atomicity**: How does PHP handle concurrent bets/credits?
   - Database transaction strategy
   - Locking mechanism (pessimistic or optimistic)
   - Rollback guarantees

2. **Financial Precision**:
   - Currency handling (which currencies supported?)
   - Decimal places (cents vs. smaller units?)
   - Rounding rules (for bet splits, partial wins)

3. **Idempotency**:
   - How are duplicate webhooks detected?
   - Is there a transaction ID/reference used?
   - How long are duplicate windows?

4. **Session Binding**:
   - How is player tied to operator?
   - Can player have multiple concurrent sessions?
   - Session timeout behavior?

### Operator Context Dependencies
Evolution likely requires strict operator isolation:
- Each bet must belong to specific operator
- Balance checks must be operator-scoped
- Rollback must only affect specific operator's transactions

### Go Implementation Considerations
- Transaction state machine must be explicit in Go (not implicit in ORM)
- GORM hooks for transaction safety
- Concurrency patterns (channels, mutex, atomic operations)
- Idempotency key tracking in database
